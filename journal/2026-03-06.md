## Technical Accomplishments

### 1. The $100 Cap Bug: Fixing Proportional Kelly Sizing

The day began by diagnosing why every paper trade position was landing at approximately $100 regardless of edge magnitude. The root cause was `max_position_dollars: 100` in `config/settings.yaml` -- a hard cap that triggered on **every** trade because even the weakest-edge trades (8%) produced Kelly-optimal dollar amounts of ~$240, well above $100.

**The math before the fix:**

| Edge | Full Kelly | Frac Kelly (15%) | Dollar Amt | After $100 Cap |
|------|-----------|-----------------|-----------|----------------|
| 63%  | 86.3%     | 12.95%          | $1,295    | **$100**       |
| 20%  | 36.3%     | 5.45%           | $545      | **$100**       |
| 8%   | 16.0%     | 2.4%            | $240      | **$100**       |

Every trade hit the same ceiling. Kelly sizing was effectively disabled.

**The fix:** Raised `max_position_dollars` from $100 to $1,000 and `max_position_pct` from 5% to 10%. Now there is a genuine 4x range between high-edge and low-edge position sizes:

| Edge | Frac Kelly $ | After 10% Cap | After $1k Cap | Effective Cost |
|------|-------------|--------------|--------------|----------------|
| 63%  | $1,295      | $1,000       | $1,000       | **$1,000**     |
| 20%  | $545        | $545         | $545         | **$545**       |
| 8%   | $240        | $240         | $240         | **$240**       |

**Files changed:** `config/settings.yaml` (lines 23-24), `src/config/settings.py` (lines 55-56)

### 6. Strategy Management System (Phase 2)

Built a complete strategy management system enabling named strategy configurations to be created, saved, edited, activated, and deleted.

**Database schema:** New `strategies` table with 17 columns covering:
- Entry filters: `edge_threshold`, `min_edge_hrrr_confirm`, `min_model_prob`
- Position sizing: `fractional_kelly`, `max_position_pct`, `max_position_dollars`
- Risk management: `daily_loss_limit`, `max_concurrent_positions`, `max_positions_per_city`, `max_positions_per_date`
- Market filters: `min_price`, `max_price`, `max_lead_hours`
- Metadata: `is_active`, `created_at`, `updated_at`

**Trade tagging:** Added `strategy_id` foreign key to the `trades` table (nullable, backward-compatible with existing trades).

**API endpoints (7 new routes):**
- `GET /api/strategies` -- list all strategies
- `POST /api/strategies` -- create new strategy
- `GET /api/strategies/:id` -- get single strategy
- `PUT /api/strategies/:id` -- update strategy parameters
- `DELETE /api/strategies/:id` -- delete strategy
- `POST /api/strategies/:id/activate` -- set as active for bot
- `GET /api/strategies/:id/trades` -- get strategy-specific trades
- `GET /api/strategies/:id/status` -- strategy performance metrics

**Bot integration:** `_load_active_strategy()` in `src/scheduler/jobs.py` queries the active strategy from the DB on each scan cycle, overrides `StrategyConfig` fields, and rebuilds the `EdgeDetector`, `KellySizer`, and `RiskManager` instances. Trades are tagged with `strategy_id`.

**Lightweight migration:** The API uses `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ADD COLUMN` checks on startup, avoiding the need for a formal migration framework (Alembic) at this stage.

**Files created/modified:** `src/data/models.py` (new Strategy, BacktestRun, OptimizationRun classes), `dashboard/app.py` (7 endpoints + table creation helpers), `src/scheduler/jobs.py` (`_load_active_strategy()`)

### 7. Per-Strategy Backtesting (Phase 3)

Wrapped the existing `BacktestEngine` to accept strategy parameters from the database and return API-friendly results.

**Strategy runner** (`src/backtest/strategy_runner.py`, 122 lines):

### 8. Parameter Optimization Lab (Phase 4)

Created a grid search optimizer that runs backtests across all combinations of parameter values.

**Optimizer** (`src/backtest/optimizer.py`, 113 lines):
- `BacktestOptimizer.grid_search()` accepts parameter ranges as `{param_name: [values]}` dicts
- Uses `itertools.product` to enumerate all combinations
- Runs `run_strategy_backtest()` for each combination
- Returns sorted results with metrics for each parameter set

**Supported sweep parameters:**
- `edge_threshold`: e.g. [0.04, 0.06, 0.08, 0.10, 0.12]
- `min_model_prob`: e.g. [0.50, 0.55, 0.60, 0.65]
- `fractional_kelly`: e.g. [0.10, 0.15, 0.20, 0.25]
- `max_position_pct`: e.g. [0.05, 0.10, 0.15]
- `max_lead_hours`: e.g. [24, 48, 72]

**Results storage:** New `optimization_runs` table with JSON-serialized parameter ranges, full results array, and best parameter set.

**API endpoints (4 new routes):**
- `POST /api/optimize/run` -- run grid search (synchronous)
- `GET /api/optimize/runs` -- list optimization runs
- `GET /api/optimize/runs/:id` -- full results
- `POST /api/optimize/runs/:id/create-strategy` -- create strategy from best params

## Architecture Decisions

### Why lightweight SQLite migrations instead of Alembic?

The project is in early paper-trading phase with a single developer. Using `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ADD COLUMN` checks on dashboard startup gives us the schema evolution we need without adding Alembic as a dependency. Once the schema stabilizes and we approach live trading, migrating to Alembic would be warranted.

### Why synchronous backtest API calls?

The backtest engine runs in 5-30 seconds for typical date ranges. Making it async would add complexity (WebSocket progress, polling, background workers) without meaningful UX benefit. The optimization grid search can take longer (30s-5min) but remains synchronous with a loading spinner for now. If grid sizes grow beyond ~100 combinations, async with progress reporting would be the next evolution.

### Why no SQLAlchemy in the dashboard?

The Flask dashboard uses raw `sqlite3` connections rather than the SQLAlchemy ORM used by the bot core. This is intentional -- the dashboard is a lightweight monitoring layer that doesn't need the ORM's abstraction overhead. The `_ensure_*_table()` functions provide schema auto-creation without pulling in the full `src.data.db` machinery.

### Why strategy override per-scan instead of persistent config?

The bot loads the active strategy's parameters from the DB on every 5-minute scan cycle rather than caching them. This means parameter changes take effect within 5 minutes without restarting the bot. The DB read is trivial (single-row lookup) and the overhead is negligible.
